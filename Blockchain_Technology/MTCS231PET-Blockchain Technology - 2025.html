A general cubic polynomial x³ + ax + b = 0 can have up to three distinct real roots (or one real + two complex conjugate roots). In blockchain Elliptic Curve Cryptography (ECC), the curve is defined as y² = x³ + ax + b and must satisfy a non-singularity condition (4a³ + 27b² ≠ 0), which guarantees the curve has well-defined distinct roots/points for the group law.

Q 2 Prove 242³²⁹ mod 243 using Fermat's Theorem (p=13, a=11) 6 marks
Answer

Fermat's Little Theorem

If p is a prime number and a is an integer not divisible by p, then:

ap−1 ≡ 1 (mod p)

Part A — Compute 242³²⁹ mod 243

Observe: 243 = 3⁵ (NOT a prime number)
Also: 242 = 243 − 1 ≡ −1 (mod 243)
Therefore: 242³²⁹ ≡ (−1)³²⁹ (mod 243)
Since 329 is ODD: (−1)³²⁹ = −1 ≡ 242 (mod 243)
✓ 242³²⁹ mod 243 = 242

Part B — Verify Fermat's Theorem with p = 13, a = 11

We need to show that: 11¹² ≡ 1 (mod 13)

11¹ mod 13 = 11
11² mod 13 = 121 mod 13 = 121 − 9×13 = 121 − 117 = 4
11³ mod 13 = 11 × 4 = 44 mod 13 = 44 − 3×13 = 44 − 39 = 5
11⁴ mod 13 = 11 × 5 = 55 mod 13 = 55 − 4×13 = 55 − 52 = 3
11⁶ mod 13 = (11³)² mod 13 = 5² mod 13 = 25 mod 13 = 12 ≡ −1
11¹² mod 13 = (11⁶)² mod 13 = (−1)² mod 13 = 1
✓ 11¹² ≡ 1 (mod 13) — Fermat's Theorem VERIFIED
Why this matters in Blockchain: Fermat's Little Theorem is the mathematical foundation of RSA encryption and modular arithmetic used in generating public-private key pairs, which secure every wallet and transaction in blockchain.
Q 3 Diffie-Hellman Key Exchange Algorithm and Problems 6 marks
Answer

What is Diffie-Hellman?

The Diffie-Hellman (DH) key exchange is a method that allows two parties to securely establish a shared secret key over an insecure channel — without ever sending the key itself.

Problems with Diffie-Hellman

ProblemExplanation
Man-in-the-Middle AttackAn attacker can intercept and replace both public keys. DH alone provides no authentication, so neither party knows WHO they're talking to.
Discrete LogarithmSecurity relies on the difficulty of computing discrete logs. With quantum computers, this could be broken (Shor's algorithm).
No AuthenticationDH is only a key-exchange protocol, not a full security system. It must be combined with digital signatures (e.g., RSA, ECDSA).
In blockchain, DH principles underpin protocols like ECDH (Elliptic Curve DH), used for secure wallet key agreement and encrypted messaging between nodes.
Q 4 Blockchain Fork – Definition, Types, Why Forking? 6 marks
Answer

What is a Blockchain Fork?

A Fork is a change to the blockchain's protocol (rules). When some nodes adopt a new rule set while others don't, the blockchain can "split" into two separate chains — like a fork in a road.

Types of Forks

TypeBackward Compatible?ResultExample
Hard ForkNo — all nodes must upgradePermanent split into two separate chainsBitcoin → Bitcoin Cash (2017)
Soft ForkYes — new rules stricter; old nodes still accept new blocksChain converges; old nodes continue (partly)SegWit upgrade in Bitcoin

Why is Forking Necessary?

  • Protocol upgrades: Fix bugs, improve efficiency, add features
  • Scaling: Increase block size or transaction throughput
  • Governance disputes: Community disagreements on direction (e.g., block size wars)
  • Security patches: Emergency fixes to critical vulnerabilities
  • Community splits: Create new cryptocurrency from an existing one
Q 5 Merkle Tree – Key Components and Importance in Blockchain 6 marks
Answer

What is a Merkle Tree?

A Merkle Tree is a binary hash tree where every leaf node is a hash of a transaction, and every non-leaf node is a hash of its two children. The topmost node is called the Merkle Root.

Key Components

ComponentDescription
Leaf NodeHash of an individual transaction (SHA-256 in Bitcoin)
Branch/Internal NodeHash of the concatenation of its two children
Merkle RootSingle top-level hash that represents ALL transactions in a block
Merkle Proof/PathMinimum set of hashes needed to verify a single transaction

Why is Merkle Tree Important in Blockchain?

  • Efficient verification: To prove Tx3 is in the block, you only need Hash(Tx4) and Hash-12 — just 2 hashes instead of all 4 transactions. For 1 million transactions, only ~20 hashes needed.
  • Tamper detection: Changing any single transaction changes its leaf hash, which propagates up and changes the Merkle Root — immediately detectable.
  • Light clients (SPV): Mobile wallets download only block headers (80 bytes with Merkle Root) and verify transactions with a Merkle Proof, without downloading the full 1 MB block.
  • Data integrity: The Merkle Root stored in the block header acts as a fingerprint of all transactions.
Q 6 Proof-of-Work (PoW) – What it is and How it Works 6 marks
Answer

What is Proof-of-Work?

Proof-of-Work (PoW) is a consensus mechanism that requires participants (miners) to solve a computationally difficult mathematical puzzle to add a new block to the blockchain. It is "proof" that real computational work was done.

How PoW Works (Step by Step)

Concrete Example Target: find nonce such that SHA-256(Block Data + Nonce) starts with "0000..."
Nonce = 0 → hash = a4f2c... (FAIL)
Nonce = 1 → hash = b2e8d... (FAIL)
Nonce = 48271 → hash = 0000a3f... (SUCCESS! Block added.)

Properties of PoW

  • Hard to compute, easy to verify: Takes miners huge effort to find nonce, but anyone can verify the hash in milliseconds
  • Difficulty adjustment: Bitcoin adjusts difficulty every 2016 blocks (~2 weeks) to maintain ~10-minute block time
  • 51% attack prevention: To attack, you'd need >50% of all mining power in the world
  • Drawback: Enormous energy consumption (Bitcoin uses ~150 TWh/year, comparable to Argentina)
Q 7 Mempool Purpose & Role of SegWit in Blockchain 6 marks
Answer

Mempool (Memory Pool)

The Mempool is a node's local waiting area for unconfirmed transactions. When you send Bitcoin, your transaction first goes into the mempool before a miner selects it to include in a block.

AspectDetail
PurposeTemporary buffer holding broadcast transactions awaiting confirmation
PriorityMiners pick transactions with highest fee-per-byte first
ClearingTransaction leaves mempool once included in a mined block
CongestionDuring peak demand (e.g., NFT craze), mempool can hold 100,000+ transactions causing fee spikes

Segregated Witness (SegWit)

SegWit was a soft fork activated on Bitcoin in August 2017. It changes how transaction data is structured to solve two major problems.

What SegWit Solves

  • Transaction Malleability Bug: Signatures (witness data) were part of the transaction ID, allowing attackers to modify signatures and change the TxID before confirmation. SegWit separates witness data, fixing this bug — essential for Lightning Network.
  • Block capacity increase: By moving witness data outside the 1 MB limit (using block weight: 1 byte data = 4 weight units, 1 byte witness = 1 weight unit), effective capacity increased to ~1.7–2 MB per block.
  • Fee reduction: Users pay less because signature data is cheaper per weight unit.

Ethereum Virtual Machine (EVM)

The EVM is the runtime environment for smart contracts on Ethereum. It is a sandboxed, deterministic virtual machine that runs on every Ethereum node.

Why EVM is important:
• Deterministic: Same code + same input = same output on ALL nodes worldwide. This ensures consensus.
• Isolated (sandboxed): Smart contract code cannot access the host computer's files or network — it can only interact with blockchain state.
• Turing-complete: Can execute any computable function (loops, conditionals, etc.)
• Gas metering: Every EVM opcode costs a defined amount of gas, preventing infinite loops and DDoS attacks.
• Portability: EVM-compatible blockchains (Polygon, BSC, Avalanche) can run Ethereum smart contracts without modification.
How EVM executes a transaction 1. User sends ETH to a contract address with calldata
2. EVM loads the contract's bytecode
3. EVM executes instructions (PUSH, ADD, SSTORE, etc.) one by one
4. Gas deducted per opcode
5. State changes committed to blockchain if execution succeeds; reverted if gas runs out
Q 9 Hyperledger Fabric – Transaction Flow (Sequence Diagram) 6 marks
Answer

Transaction Flow in Hyperledger Fabric

Hyperledger Fabric uses an Execute-Order-Validate architecture, unlike Bitcoin's Order-Execute model. This allows parallel execution and prevents non-determinism.

Summary of Steps

  1. Client creates a transaction proposal and sends it to endorsing peers
  2. Endorsing Peer simulates the transaction by executing chaincode (without writing to ledger)
  3. Peer sends back a Read-Write Set with its digital signature (endorsement)
  4. Client collects enough endorsements (per endorsement policy) and submits to Orderer
  5. Orderer orders all transactions, packages them into a block, and distributes to all peers
  6. Committing Peers validate endorsements and MVCC conflicts, then commit block to ledger
  7. Event notification sent back to client confirming success/failure

Q 11 RAFT Consensus – What it is, Leader Election, Failure Handling, Limitations 10 marks
Answer

What is RAFT?

RAFT (Reliable, Replicated, Redundant, And Fault-Tolerant) is a consensus algorithm designed for distributed systems to agree on a single sequence of log entries. It is simpler to understand than Paxos while providing equivalent guarantees.

RAFT is used in Hyperledger Fabric (as its Ordering Service since v1.4.1) and systems like etcd, CockroachDB, Consul.

Key Concept: Node Roles

RoleResponsibility
LeaderHandles ALL client requests; replicates log entries to followers; exactly ONE at a time
FollowerPassive; responds to requests from leaders and candidates; replicates leader's log
CandidateTemporary state during leader election; campaigns for votes

Leader Election Process

Log Replication (How RAFT maintains consistency)

  1. Client sends request to Leader
  2. Leader appends entry to its log (not yet committed)
  3. Leader sends AppendEntries RPC to all followers
  4. Once majority (quorum) acknowledges, Leader commits the entry
  5. Leader responds to client; followers commit on next heartbeat

How RAFT Handles Failures

Failure TypeRAFT's Response
Leader crashElection timeout → new leader elected from followers. System tolerates up to (N-1)/2 failures in N nodes.
Follower crashLeader keeps retrying AppendEntries. Follower catches up via log when it rejoins.
Network partitionPartition with majority elects new leader. Minority partition cannot process requests (no quorum). Old leader steps down when it discovers higher term.
Split voteRandom election timeouts reduce collision probability. A new election begins in the next term.

Limitations of RAFT

  • Crash-fault tolerant only: RAFT assumes nodes fail by crashing, not by behaving maliciously (Byzantine faults). Unsuitable for public blockchains where nodes may act adversarially.
  • Scalability: Performance degrades with many nodes due to O(N) messaging for each log entry. Works well with 3–7 orderer nodes in Hyperledger Fabric.
  • Leader bottleneck: All writes go through one leader — potential performance bottleneck.
  • Majority requirement: Needs >50% nodes alive. A 5-node cluster can only tolerate 2 simultaneous failures.

What is the "Current Target"?

The target is a 256-bit number. For a block to be valid, the SHA-256 hash of its header must be less than or equal to this target. A lower target = harder puzzle.

Decoding the Target from "Bits" Field

Bits field (hex) = 0x17318C29
Format: Bits = 0xAABBBBBB where AA = exponent, BBBBBB = coefficient
Exponent = 0x17 = 23
Coefficient = 0x318C29
Target = Coefficient × 2^(8×(Exponent − 3))
= 0x318C29 × 2^(8×20)
≈ 0x00000000000000000031 8C29 0000... (zeros padded)
Target ≈ 0x00000000000000000031... (very small number → very hard)

Where is Current Target Stored?

  • The target is NOT stored explicitly in the block
  • It is computed by all nodes from the "Bits" field in the block header
  • The Bits field is a compact encoding of the current target
  • Every Bitcoin node independently re-derives the target and verifies the block hash

Difficulty Level Explained

Genesis Block Target (Max Target) = 0x00000000FFFF × 2^208
Difficulty = Max Target / Current Target
Difficulty = 6,727,225,469,722.53
This means mining this block is ~6.7 TRILLION times harder than the Genesis block
On average, miners had to try 6.7 × 10¹² hashes to find a valid nonce

Difficulty Adjustment Algorithm

Why Difficulty Adjusts If more miners join the network (hashrate increases), blocks get found faster than 10 minutes. The difficulty increases to compensate, maintaining the 10-minute average. If miners leave, difficulty decreases. This self-regulating mechanism is crucial to Bitcoin's monetary policy (fixed block time = predictable issuance).

c. Orphaned Block

An orphaned block (also called a stale block) is a valid block that was mined at the same time as another block at the same height, but was not selected as the canonical chain's block.

Why it happens: Two miners solve the puzzle almost simultaneously and broadcast their blocks. The network temporarily has two valid chains. The "longest chain rule" (most cumulative proof-of-work) resolves this — one chain gets the next block added first and becomes canonical. The other block is orphaned.

In Bitcoin: Orphaned block miners lose their block reward (12.5 BTC was wasted work).
Rate: Bitcoin has ~0.2% orphan rate. Higher in Ethereum pre-Merge (leading to Uncle blocks).

d. Double Spending

Double spending is the attempt to spend the same cryptocurrency twice. This is the fundamental problem digital money solves — unlike physical cash, digital data can be copied.

Attack Scenario 1. Mallory sends 1 BTC to Alice (for goods)
2. Simultaneously, Mallory broadcasts another transaction sending the same 1 BTC to her own address
3. With enough hashpower (51% attack), Mallory can make the second transaction the canonical one
4. Alice's goods are gone, and Mallory still has the Bitcoin

How Blockchain prevents it: Transactions are broadcast to the network, and nodes reject any UTXO (Unspent Transaction Output) that has already been spent. Waiting for 6+ confirmations makes reversal practically impossible.

e. Uncle Block (Ethereum)

Uncle blocks (called "ommer blocks" in Ethereum) are the Ethereum equivalent of Bitcoin's orphaned blocks, but with an important difference: Ethereum actually rewards miners for uncle blocks.

FeatureBitcoin Orphaned BlockEthereum Uncle Block
Included in chain?No, completely discardedReferenced (but not full canonical)
RewardNone (full loss)7/8 of block reward (~87.5%)
Security contributionNone countedIncreases total difficulty (GHOST protocol)
Max depthN/AOnly blocks within 7 generations can be uncles

This design (GHOST protocol) lets Ethereum have faster block times (13–15 seconds) while still maintaining security, since uncle blocks contribute to the chain's total work.

Q 14 Elliptic Curve Cryptography – Algorithm, Vulnerabilities & Applications 10 marks
Answer

What is Elliptic Curve Cryptography (ECC)?

ECC is a public-key cryptography approach based on the algebraic structure of elliptic curves over finite fields. It provides equivalent security to RSA with much smaller key sizes.

ECC Algorithm for Encryption/Decryption (ECIES)

To Encrypt message M (sender uses receiver's public key Q):

  1. Choose random number r
  2. Compute R = r × G (ephemeral public key)
  3. Compute shared secret S = r × Q (using receiver's public key)
  4. Derive symmetric key from S (using KDF)
  5. Encrypt M with symmetric key → Ciphertext C
  6. Send (R, C) to receiver

To Decrypt (receiver uses private key d, where Q = d × G):

  1. Compute S = d × R = d × r × G = r × d × G = r × Q (same shared secret!)
  2. Derive symmetric key from S
  3. Decrypt C to get M

Key Size Comparison

AlgorithmKey Size for 128-bit SecurityPerformance
RSA3072 bitsSlow (large exponentiation)
ECC256 bitsFast (12× faster than RSA)
AES (symmetric)128 bitsFastest

Vulnerabilities of ECC

VulnerabilityDescription
Quantum Computing (Shor's Algorithm)A sufficiently powerful quantum computer can solve the Elliptic Curve Discrete Logarithm Problem (ECDLP) in polynomial time, breaking ECC. Post-quantum alternatives (lattice-based crypto) are being standardized by NIST.
Invalid Curve AttackIf implementation doesn't validate that the point is actually on the curve, an attacker can use a point on a weaker curve to extract the private key.
Weak Random Number GenerationIn ECDSA, if the random nonce k is reused or predictable, the private key can be directly computed. (This broke the Sony PS3's code-signing in 2010.)
Side-Channel AttacksTiming attacks, power analysis on hardware implementations can leak private key bits during scalar multiplication.
Backdoored CurvesDual_EC_DRBG (an NSA-proposed RNG using ECC) was found to contain a backdoor. Trustworthy curves (secp256k1, Curve25519) are chosen for transparent parameter selection.

Applications of ECC in Blockchain

  • Bitcoin/Ethereum wallets: secp256k1 curve generates key pairs. Private key = 256 bits; public key = 512 bits (compressed to 264 bits).
  • ECDSA (Digital Signatures): Every Bitcoin transaction is signed with ECDSA to prove ownership without revealing private key.
  • Ethereum addresses: Address = last 20 bytes of Keccak-256 hash of public key (derived via ECC).
  • ECDH Key Exchange: Used in blockchain P2P network encryption for secure communication between nodes.
  • EdDSA / Ed25519: Used in newer blockchains (Solana, Cardano) for faster signature verification (deterministic nonce — no weak-k problem).
  • Zero-Knowledge Proofs: ECC arithmetic forms the basis of many ZK proof systems (zk-SNARKs use BLS12-381 curve).
