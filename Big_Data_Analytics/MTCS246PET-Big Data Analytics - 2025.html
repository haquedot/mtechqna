Q2

MapReduce Phases & Data Aggregation

Answer: Data Aggregation in MapReduce happens in the Reduce Phase (Option C).

What is MapReduce?

Definition MapReduce is a programming model used to process large datasets in parallel across a Hadoop cluster. It breaks a big job into two main functions: Map (split and process) and Reduce (aggregate results).

MapReduce Phases — Step by Step

INPUT DATA | v ┌─────────────┐ │ MAP PHASE │ ← Processes each record, emits Key-Value pairs └─────────────┘ | v ┌──────────────────┐ │ SHUFFLE PHASE │ ← Groups same keys together, sorts data └──────────────────┘ | v ┌───────────────────┐ │ REDUCE PHASE │ ← DATA AGGREGATION happens here! └───────────────────┘ | v OUTPUT / HDFS
PhaseWhat HappensInputOutput
Map PhaseReads input, processes each record, emits (key, value) pairsRaw data chunksKey-Value pairs
Shuffle PhaseGroups all values with the same key together, sorts themMap outputGrouped Key-Value pairs
Reduce PhaseAggregates / summarizes grouped values per keyGrouped pairsFinal result

Why Reduce Phase = Data Aggregation?

The Reduce phase takes all values for a given key and aggregates them — it sums, counts, averages, or combines them. This is exactly what data aggregation means: combining multiple values into one summary.

Example (Word Count):

Map Output: ("hadoop",1), ("hadoop",1), ("data",1), ("data",1), ("data",1) After Shuffle: "hadoop" → [1,1] "data" → [1,1,1] Reduce Output: ("hadoop", 2) ← AGGREGATION: sum of [1,1] ("data", 3) ← AGGREGATION: sum of [1,1,1]
⭐ Key Points for Revision
  • MapReduce has 3 main phases: Map → Shuffle → Reduce
  • Data Aggregation = Reduce Phase (combines values per key)
  • Shuffle Phase sorts and groups data (transfers between Map and Reduce)
  • Map Phase does the parallel processing on input splits
  • MERGE phase is NOT a standard MapReduce phase
Conclusion In MapReduce, Data Aggregation occurs in the Reduce Phase. The Map phase processes raw input, the Shuffle phase organizes keys, and the Reduce phase aggregates all values per key to produce the final result.

Hadoop Cluster Architecture (NameNode, DataNode, etc.)

Definition A Hadoop Cluster is a collection of computers (nodes) that work together to store and process big data. It follows a Master-Slave architecture where the Master manages resources and Slaves do the actual work.

Hadoop Cluster Architecture Diagram

┌─────────────────────────────────┐ │ MASTER NODE │ │ ┌───────────┐ ┌─────────────┐ │ │ │ NameNode │ │ResourceMgr │ │ │ │ (HDFS) │ │ (YARN) │ │ │ └───────────┘ └─────────────┘ │ └──────────────┬──────────────────┘ │ ┌───────────────────┼───────────────────┐ │ │ │ ┌──────────┴──────┐ ┌─────────┴──────┐ ┌────────┴───────┐ │ SLAVE NODE 1 │ │ SLAVE NODE 2 │ │ SLAVE NODE 3 │ │ ┌──────────┐ │ │ ┌──────────┐ │ │ ┌──────────┐ │ │ │DataNode │ │ │ │DataNode │ │ │ │DataNode │ │ │ └──────────┘ │ │ └──────────┘ │ │ └──────────┘ │ │ ┌──────────┐ │ │ ┌──────────┐ │ │ ┌──────────┐ │ │ │Node Mgr │ │ │ │Node Mgr │ │ │ │Node Mgr │ │ │ └──────────┘ │ │ └──────────┘ │ │ └──────────┘ │ └─────────────────┘ └────────────────┘ └────────────────┘

Key Components Explained

1. NameNode (Master – HDFS)

  • Stores metadata only (file names, directory structure, block locations)
  • Does NOT store actual data
  • There is only one NameNode per cluster (Single Point of Failure → solved by Secondary NameNode)
  • Manages the file system namespace

2. DataNode (Slave – HDFS)

  • Stores the actual data blocks
  • Sends heartbeat signals to NameNode every 3 seconds
  • Multiple DataNodes can exist in a cluster
  • Data is replicated across DataNodes (default replication factor = 3)

3. Secondary NameNode

  • NOT a backup NameNode — it's a checkpoint node
  • Merges the edit log with the FsImage to prevent NameNode from getting too large

4. ResourceManager (Master – YARN)

  • Manages resource allocation across the cluster
  • Schedules jobs and allocates containers

5. NodeManager (Slave – YARN)

  • Runs on each slave node
  • Monitors CPU, memory, disk usage of containers
  • Reports to ResourceManager

⭐ Key Points for Revision
  • Hadoop uses Master-Slave architecture
  • NameNode = metadata; DataNode = actual data blocks
  • Secondary NameNode ≠ Backup NameNode; it does checkpointing
  • Default block size in HDFS = 128 MB
  • Default replication factor = 3 (each block stored on 3 DataNodes)
  • HDFS answer for the MCQ: only one NameNode per cluster (answer: D)
Conclusion Hadoop Cluster uses a Master-Slave model. The NameNode and ResourceManager are master components that manage metadata and resource scheduling. DataNodes and NodeManagers are slave components that store data and execute tasks.
Q4

HDFS & Write-Once, Read-Many Principle

Answer: TRUE. HDFS follows the Write-Once, Read-Many (WORM) model.
Definition: HDFS Hadoop Distributed File System (HDFS) is the storage layer of Hadoop. It stores very large files across multiple machines and provides high throughput access to data. It is designed for commodity hardware.

Write-Once, Read-Many (WORM) Principle

In HDFS, once a file is written, it cannot be modified. You can only:

  • Append to end of file (limited)
  • Read the file multiple times
  • Delete the file

This is different from regular file systems where you can edit files at any point.

Why WORM? Reasons:

  • Simplicity: No complex locking mechanisms needed
  • Performance: Sequential reads are faster than random access
  • Fault Tolerance: Replicated blocks don't need to handle concurrent writes
  • Batch Processing: Big Data workloads are mostly read-heavy analytics jobs

HDFS vs NTFS Comparison

⭐ Key Points for Revision
  • HDFS = Write-Once, Read-Many (TRUE — it's a core design principle)
  • Files cannot be randomly modified after writing
  • HDFS block size = 128 MB (older versions: 64 MB)
  • Each block is replicated to 3 DataNodes by default
  • Suitable for streaming data access (batch analytics), not interactive apps
  • HDFS works best with large files — many small files are inefficient
Conclusion HDFS follows the Write-Once, Read-Many principle to ensure high throughput, fault tolerance, and simplicity. It is optimized for large sequential reads, making it ideal for Big Data analytics where data is written once and analyzed many times.

NoSQL Databases — Definition, Types & Examples

Answer for MCQ: MongoDB (Option C) is the most commonly used NoSQL database for handling Big Data.
Definition: NoSQL NoSQL (Not Only SQL) is a type of database that does not use the traditional table-based relational structure. It is designed to store, retrieve, and manage unstructured or semi-structured data at very large scales with high performance.

Why NoSQL Over Traditional SQL?

  • Handles unstructured data (social media posts, images, logs)
  • Scales horizontally (add more servers, not bigger servers)
  • High availability with no single point of failure
  • Flexible schema — no predefined table structure needed
  • Faster read/write for large volumes

Types of NoSQL Databases

1. Key-Value Store

  • Data stored as simple key → value pairs
  • Fastest type of NoSQL
  • Examples: Redis, DynamoDB, Riak
  • Use case: Shopping cart, session management, caching

2. Document Store

  • Data stored as JSON/BSON documents
  • Each document can have different fields (flexible schema)
  • Examples: MongoDB, CouchDB
  • Use case: Content management, user profiles, catalogs

3. Column-Family Store

  • Data stored in columns instead of rows
  • Very fast for reading specific columns of large datasets
  • Examples: Apache Cassandra, HBase
  • Use case: IoT data, time-series data, sensor readings

4. Graph Database

  • Data stored as nodes and edges (relationships)
  • Best for highly connected data
  • Examples: Neo4j, Amazon Neptune
  • Use case: Social networks, fraud detection, recommendation engines
TypeData ModelBest ExampleUse Case
Key-Valuekey → valueRedisCaching, Sessions
DocumentJSON documentsMongoDBCMS, Catalogs
ColumnColumn familiesCassandraIoT, Analytics
GraphNodes + EdgesNeo4jSocial Networks

MongoDB Example (Document):

{ "_id": "001", "name": "Ali Hassan", "age": 28, "courses": ["Big Data", "Machine Learning"], "address": { "city": "Hyderabad", "state": "Telangana" } } ← No fixed schema! Fields can vary per document.
⭐ Key Points for Revision
  • NoSQL = Not Only SQL — it INCLUDES SQL-like querying in some DBs
  • 4 types: Key-Value, Document, Column-Family, Graph
  • MongoDB is document-based and most widely used for Big Data
  • NoSQL is horizontally scalable (add more nodes)
  • NoSQL follows BASE model: Basically Available, Soft state, Eventual consistency
  • SQL follows ACID: Atomicity, Consistency, Isolation, Durability
Conclusion NoSQL databases provide flexibility, scalability, and performance for Big Data workloads. MongoDB is the most popular NoSQL database for handling large volumes of unstructured data.

Oozie — Definition and Role in Hadoop

Definition: Oozie Oozie is a workflow scheduling system for Hadoop. It is used to manage and coordinate multiple Hadoop jobs (MapReduce, Hive, Pig, HDFS, Sqoop) and execute them in the correct order based on time or data availability.

What Problem Does Oozie Solve?

In real Big Data pipelines, you don't run just one job. You run a sequence of jobs:

Ingest data (Sqoop)
→
Clean data (Pig)
→
Analyze (Hive)
→
Export results

Oozie manages this entire pipeline automatically.

Types of Oozie Jobs

1. Workflow Jobs

  • Defines a Directed Acyclic Graph (DAG) of actions
  • Actions run in sequence: one after another
  • Defined in XML format

2. Coordinator Jobs

  • Schedule workflow jobs at specific times
  • Like a cron job for Hadoop
  • Example: Run ETL job every day at midnight

3. Bundle Jobs

  • Group of coordinator jobs managed together
  • Used for complex data pipelines

Oozie Workflow Diagram

START | v [Action 1: Sqoop Import] | v [Action 2: MapReduce Job] | / \ / \ OK FAIL | | v v [Action 3] [Error Handler] | v END

Real-Life Example:

An e-commerce company uses Oozie to:

  • Import orders from MySQL to HDFS using Sqoop (12 AM)
  • Process orders with MapReduce (1 AM)
  • Generate daily sales report using Hive (2 AM)
  • Send report to dashboard (3 AM)
⭐ Key Points for Revision
  • Oozie = Hadoop workflow scheduler and coordinator
  • Written in Java; web-based UI also available
  • Works as a server running inside a Java Servlet container
  • 3 types: Workflow, Coordinator, Bundle
  • Workflows are defined in XML (hPDL — Hadoop Process Definition Language)
  • Oozie is NOT a standard Hadoop component — it is a tool in the Hadoop Ecosystem
Conclusion Oozie is a critical workflow scheduler in the Hadoop ecosystem that automates and coordinates sequences of data processing jobs. It allows Big Data engineers to build reliable data pipelines with scheduled execution and error handling.

Characteristics of Big Data — The 5 V's

Answer for MCQ: "High volume" (high, not low), "Unstructured format", "High velocity" and "Unpredictable variety" are characteristics of Big Data. Option A (Low volume) is WRONG — Big Data has HIGH volume.
Definition: Big Data Big Data refers to datasets that are so large, fast-moving, and complex that traditional databases cannot handle them. It is characterized by the 5 V's: Volume, Velocity, Variety, Veracity, and Value.

The 5 V's of Big Data

📦
Volume
Massive amounts of data (Terabytes to Zettabytes). Ex: 2.5 quintillion bytes created daily.
⚡
Velocity
Speed at which data is generated. Ex: Twitter generates 500M tweets/day.
🗂️
Variety
Different types: structured, unstructured, semi-structured. Ex: text, video, JSON, CSV.
✅
Veracity
Accuracy and trustworthiness of data. Ex: noisy sensor data must be cleaned.
💎
Value
Business insights derived from data. Ex: Amazon uses purchase data to increase sales by 35%.

Detailed Explanation

1. Volume — "How much?"

  • Google processes 40,000 searches per second
  • Facebook generates 4 petabytes of data daily
  • Traditional databases fail at this scale — Hadoop handles it with distributed storage

2. Velocity — "How fast?"

  • Real-time data streams: stock market, sensor data, social media
  • Requires streaming frameworks like Apache Kafka and Spark Streaming

3. Variety — "What types?"

  • Structured: Tables in databases (rows and columns)
  • Semi-structured: JSON, XML, CSV files
  • Unstructured: Videos, images, emails, social media posts (80% of all data!)

4. Veracity — "How reliable?"

  • Data can be noisy, incomplete, inconsistent
  • Requires data quality and cleansing processes

5. Value — "Why does it matter?"

  • The whole point of Big Data — extracting actionable insights
  • Only ~10% of collected data is ever analyzed

  • Big Data = HIGH Volume, HIGH Velocity, HIGH Variety (NOT low)

Nature of Data & Applications — Data Serialization

Definition: Data Serialization Data Serialization is the process of converting data structures or objects into a format that can be stored or transmitted, and then reconstructed later. It's like packing your belongings into boxes to move — you pack (serialize) and unpack (deserialize).

Types of Data by Structure

1. Structured Data

  • Organized in a predefined format (rows and columns)
  • Stored in relational databases
  • Easy to search and analyze
  • Examples: Employee table, bank transactions, spreadsheets

2. Semi-Structured Data

  • Has some organization but not in rigid table format
  • Uses tags or key-value pairs to separate elements
  • Examples: JSON files, XML documents, CSV with headers, emails

3. Unstructured Data

  • No predefined format or schema
  • Most of the world's data (80%+) is unstructured
  • Requires AI/ML to extract insights
  • Examples: Social media posts, images, videos, audio, PDFs

Data Ingestion

Collect data from multiple sources using Sqoop (databases), Flume (logs), Kafka (streams) and store in HDFS.

3

Real-Life Example: Yahoo!

Yahoo! was one of the first companies to use Hadoop at scale — running one of the world's largest Hadoop clusters with over 42,000 nodes to process petabytes of web clickstream data for advertising optimization.

⭐ Key Points for Revision
  • Hadoop was created by Doug Cutting and Mike Cafarella in 2006 (named after a toy elephant)
  • Inspired by Google's GFS and MapReduce papers (2003-2004)
  • Runs on Java — written in Java, runs on JVM
  • Commodity hardware = cheap servers (~$1000-2000 each) vs. mainframes (~$1M+)
  • Data locality: process data WHERE it is stored (move computation, not data)
  • Apache Hadoop is FREE and open-source
Conclusion Hadoop provides a complete framework for Big Data through its four core components. Its key advantage is processing data locally on each node (data locality), which avoids expensive network transfers and enables massive parallel processing.

Sqoop, Flume, and Oozie — Detailed Explanation

1. Apache Sqoop

Definition: Sqoop Sqoop (SQL-to-Hadoop) is a tool for efficiently transferring bulk data between relational databases (MySQL, Oracle, PostgreSQL) and Hadoop HDFS. It works bidirectionally — Import to HDFS and Export from HDFS.

How Sqoop Works:

Sqoop Commands:

# Import from MySQL to HDFS sqoop import --connect jdbc:mysql://localhost/company --table employees --target-dir /hdfs/employees # Export from HDFS to MySQL sqoop export --connect jdbc:mysql://localhost/company --table employees_result --export-dir /hdfs/results

Key Features:

  • Parallel import using multiple mappers (default 4)
  • Supports incremental imports (only new data)
  • Can import to Hive tables directly

2. Apache Flume

Definition: Flume Apache Flume is a distributed service for collecting, aggregating, and moving large amounts of log/event data from various sources (web servers, applications) into HDFS in real-time.

  • Sqoop uses JDBC drivers to connect to databases
  • Flume Agent = Source + Channel + Sink (one agent per node)
  • They are all part of Hadoop Ecosystem, not core Hadoop
Conclusion Sqoop, Flume, and Oozie are complementary tools in the Hadoop ecosystem. Sqoop moves batch data from/to databases, Flume streams real-time log data, and Oozie coordinates the entire workflow pipeline.

Requirements for Hadoop Framework

  • Fault Tolerance: System must continue working even when some nodes fail
  • Scalability: Easily add more nodes as data grows
  • High Throughput: Process large amounts of data quickly
  • Data Locality: Move computation to data, not data to computation
  • Commodity Hardware: Run on inexpensive standard servers
  • Open Source: Free, community-supported software
  • Reliability: Data must be safely replicated and recoverable

Hadoop Design Principles

1. Data Locality Principle

  • Move the computation (code) to the data, not the data to computation
  • Reduces network bandwidth usage drastically
  • Example: If data is on Node 3, run the MapReduce task on Node 3

2. Fault Tolerance Principle

  • Assume hardware WILL fail — design for it
  • Data is replicated to 3 nodes by default
  • Failed tasks are automatically re-run on another node

3. Simple Consistency Model

  • Write-Once, Read-Many (WORM) simplifies consistency
  • No complex lock management needed

4. Horizontal Scalability

  • Add more nodes instead of upgrading existing ones
  • Linear scaling — 2x nodes = 2x capacity (roughly)

5. Schema-on-Read

  • Data schema is applied when reading, not when writing
  • Allows storing any data first, then deciding the format later
⭐ Key Points for Revision
  • Hadoop Ecosystem has 15+ tools (Hive, Pig, HBase, Sqoop, Flume, Oozie, Zookeeper, Kafka, etc.)
  • Core design principle: Data Locality (bring code to data)
  • Hadoop is designed to handle hardware failures gracefully
  • Schema-on-read = flexibility; Schema-on-write = RDBMS
  • YARN enables multiple processing engines on same cluster (MapReduce, Spark, etc.)
Conclusion The Hadoop Ecosystem provides a complete solution for Big Data by combining specialized tools for ingestion, storage, processing, and analysis. Its design principles of data locality, fault tolerance, and horizontal scalability make it the gold standard for Big Data infrastructure.

Data Serialization — Types, Formats & Comparison

Definition: Data Serialization Serialization is the process of converting an object or data structure into a byte stream so it can be stored on disk or sent over a network. Deserialization is the reverse — converting bytes back to the original object.

Why Serialization Matters in Hadoop?

Scalability, Fault Tolerance & Efficient Storage in Hadoop

1. Scalability in Hadoop

Definition: Scalability Scalability is the ability of Hadoop to handle increasing amounts of data or jobs by adding more nodes to the cluster without changing the existing infrastructure.

Types of Scaling:

TypeMethodHadoop ApproachCost
Vertical Scaling (Scale Up)Add more CPU/RAM to existing serverNOT Hadoop's approachVery Expensive
Horizontal Scaling (Scale Out)Add more nodes to clusterHadoop's primary methodAffordable
  • Hadoop started with 2 nodes — Facebook's cluster grew to 4,000+ nodes
  • Adding nodes is transparent to the application
  • YARN automatically discovers and uses new nodes

2. Fault Tolerance in Hadoop

Definition: Fault Tolerance Fault Tolerance is Hadoop's ability to continue working correctly even when some hardware or software components fail, without losing data or stopping jobs.

Fault Tolerance Mechanisms:

1
Data Replication

Every block is stored on 3 different DataNodes. If one fails, data is still available on 2 others.

2
Heartbeat Mechanism

DataNodes send heartbeat to NameNode every 3 seconds. If heartbeat stops → node is marked dead → re-replication begins.

3
Task Re-execution

If a MapReduce task fails, YARN automatically reruns it on a healthy node. Tasks have retry limits (default 4 retries).

4
Speculative Execution

If a task is running slow, YARN launches a duplicate "speculative" task on another node. Whoever finishes first wins.

FAULT TOLERANCE EXAMPLE: Block A → DataNode 1, DataNode 2, DataNode 3 (3 replicas) DataNode 2 FAILS! NameNode detects: heartbeat from DataNode 2 stopped Action: Copy Block A to DataNode 4 (new replica) Result: Block A → DataNode 1, DataNode 3, DataNode 4 DATA NEVER LOST!

3. Efficient Storage in HDFS

How HDFS Ensures Efficient Storage HDFS uses block-based storage (128 MB blocks), compression, and intelligent replication placement to maximize storage efficiency.
  • Large Block Size (128 MB): Reduces NameNode metadata overhead; fewer blocks to manage
  • Compression: Snappy, GZip, LZO compression reduces storage by 50-75%
  • Data Locality: Keeps data close to compute; reduces network I/O
  • Rack Awareness: Replicas placed on different racks to survive rack failures
  • Columnar Formats (ORC/Parquet): Store only needed columns; reduce I/O for queries
⭐ Key Points for Revision
  • Scalability in Hadoop = Horizontal (add nodes, not upgrade existing)
  • Fault Tolerance: 3-replica rule + heartbeat + task re-execution
  • Heartbeat: every 3 seconds from DataNode to NameNode
  • Speculative execution: runs duplicate slow tasks on fresh nodes
  • Rack Awareness: 1st replica on same node, 2nd on different rack, 3rd on different rack
  • Replication factor can be changed per file with: hdfs dfs -setrep
Conclusion Hadoop's horizontal scalability allows cost-effective growth, its fault tolerance mechanisms ensure zero data loss even during hardware failures, and its block-based storage with compression ensures efficient use of disk space.
Q16

MapReduce — Word Count Problem (Step-by-Step)

Definition: MapReduce MapReduce is a parallel processing programming model in Hadoop. It divides a large job into two functions: Map (parallel processing) and Reduce (aggregation). It processes data in distributed manner across cluster nodes.

Word Count Problem

Goal: Count how many times each word appears in a large text file.

Input: "hadoop is great hadoop is fast"

Step-by-Step Word Count Flow

Step 1: Input Splitting

Input Text: "hadoop is great hadoop is fast" HDFS splits the file into chunks (input splits): Split 1: "hadoop is great" Split 2: "hadoop is fast"

Step 2: Map Phase

Mapper 1 processes "hadoop is great": hadoop → (hadoop, 1) is → (is, 1) great → (great, 1) Mapper 2 processes "hadoop is fast": hadoop → (hadoop, 1) is → (is, 1) fast → (fast, 1)

Step 3: Combiner (Optional — Mini Reducer)

Combiner runs on each Mapper node BEFORE shuffle: Mapper 1: (hadoop,1),(is,1),(great,1) → no change (each appears once) This reduces network data transferred during shuffle.

Step 4: Shuffle & Sort Phase

All (key, value) pairs are: 1. Grouped by key 2. Sorted alphabetically After Shuffle: "fast" → [1] "great" → [1] "hadoop" → [1, 1] "is" → [1, 1]

Step 5: Reduce Phase

Reducer processes each key: "fast" → sum([1]) = 1 "great" → sum([1]) = 1 "hadoop" → sum([1, 1]) = 2 "is" → sum([1, 1]) = 2

Step 6: Final Output (written to HDFS)

Output: fast 1 great 1 hadoop 2 is 2

Complete MapReduce Diagram

Input: "hadoop is great hadoop is fast" | | (Split) ┌────┴────┐ | | Split 1 Split 2 | | [MAP 1] [MAP 2] | | (h,1)(i,1) (h,1)(i,1) (g,1) (f,1) | | └────┬────┘ | [SHUFFLE] → Sort by key + Group by key | h→[1,1] i→[1,1] g→[1] f→[1] | [REDUCE] → Sum each list | hadoop:2 is:2 great:1 fast:1 | [OUTPUT → HDFS]
PhaseInputProcessingOutput
Input SplitFull text fileSplit into chunksText chunks
MapText linesTokenize, emit (word,1)(word,1) pairs
Combiner(word,1) pairsLocal aggregationReduced pairs
ShuffleAll (word,1) pairsGroup + Sort by keyword→[1,1,1...]
Reduceword→[list of 1s]Sum the list(word, count)
OutputFinal pairsWrite to HDFSText file
⭐ Key Points for Revision
  • Map function: map(key, value) → list of (key2, value2)
  • Reduce function: reduce(key2, list[value2]) → list of (key3, value3)
  • Combiner = Mini Reducer on the Mapper node (reduces network traffic)
  • Shuffle is automatic — handled by the Hadoop framework, not programmer
  • Number of Mappers = Number of Input Splits
  • Number of Reducers is configured by the programmer
Conclusion The Word Count problem illustrates how MapReduce processes data in three phases. The Map phase tokenizes input, the Shuffle phase organizes keys, and the Reduce phase aggregates counts — all running in parallel across the cluster.
Q17

YARN — Architecture, Components & Working

Definition: YARN YARN (Yet Another Resource Negotiator) is the resource management layer of Hadoop (introduced in Hadoop 2.x). It separates resource management from data processing, allowing multiple processing frameworks (MapReduce, Spark, Flink) to share the same cluster.

Why YARN was Needed?

In Hadoop 1.x, JobTracker did everything — resource management AND job scheduling. This caused a bottleneck (single point of failure and limited scalability). YARN splits these responsibilities.

Hadoop 1.x (Without YARN)Hadoop 2.x (With YARN)
JobTracker = resource manager + job schedulerResourceManager = only resource management
TaskTracker = task executorNodeManager + ApplicationMaster
Only MapReduce supportedAny framework (Spark, Tez, Flink) supported
Limited to ~4000 nodesScales to 10,000+ nodes

YARN Architecture Diagram

┌─────────────────────────────────────────────────────┐ │ MASTER NODE │ │ ┌───────────────────────────────────────────────┐ │ │ │ RESOURCE MANAGER │ │ │ │ ┌────────────────────┐ ┌──────────────────┐ │ │ │ │ │ Scheduler │ │ Applications │ │ │ │ │ │ (allocates │ │ Manager │ │ │ │ │ │ resources) │ │ (manages apps) │ │ │ │ │ └────────────────────┘ └──────────────────┘ │ │ │ └───────────────────────────────────────────────┘ │ └────────────────────────┬────────────────────────────┘ | ┌────────────────┼────────────────┐ | | | ┌───────┴──────┐ ┌───────┴──────┐ ┌───────┴──────┐ │ SLAVE NODE │ │ SLAVE NODE │ │ SLAVE NODE │ │ ┌──────────┐ │ │ ┌──────────┐ │ │ ┌──────────┐ │ │ │Node Mgr │ │ │ │Node Mgr │ │ │ │Node Mgr │ │ │ └──────────┘ │ │ └──────────┘ │ │ └──────────┘ │ │ ┌──────────┐ │ │ │ │ │ │ │App Master│ │ │ │ │ │ │ └──────────┘ │ │ │ │ │ │ ┌──┐ ┌──┐ │ │ ┌──┐ ┌──┐ │ │ ┌──┐ ┌──┐ │ │ │C1│ │C2│ │ │ │C3│ │C4│ │ │ │C5│ │C6│ │ │ └──┘ └──┘ │ │ └──┘ └──┘ │ │ └──┘ └──┘ │ │(Containers) │ │(Containers) │ │(Containers) │ └──────────────┘ └──────────────┘ └──────────────┘

YARN Components

1. ResourceManager (RM) — Master

  • Central authority for resource allocation in the cluster
  • Has two main sub-components:
    • Scheduler: Allocates resources (containers) to applications
    • Applications Manager: Manages application lifecycles (accept, schedule, monitor)

2. NodeManager (NM) — Slave

  • Runs on every slave node
  • Manages containers (CPU, memory, disk) on that node
  • Reports available resources to ResourceManager
  • Monitors container health and resource usage

3. ApplicationMaster (AM) — Per Job

  • One ApplicationMaster per submitted application/job
  • Negotiates resources (containers) from ResourceManager
  • Works with NodeManagers to execute tasks
  • Monitors task progress and restarts failed tasks

4. Container

  • A bundle of resources (CPU + Memory) allocated for a single task
  • Each container runs one Map task or one Reduce task

YARN Job Submission Flow

1
Client submits job to ResourceManager

Client sends job request, JAR file, and configuration to ResourceManager.

2
ResourceManager creates ApplicationMaster

RM allocates a container on a NodeManager and starts the ApplicationMaster in it.

3
ApplicationMaster registers with ResourceManager

AM tells RM it's alive and requests containers for Map tasks.

4
Containers allocated for Map/Reduce tasks

RM allocates containers on different NodeManagers. AM launches tasks in containers.

5
Tasks run and report progress to ApplicationMaster

Each container runs one task and reports status back to AM regularly.

6
Job completes → ApplicationMaster unregisters

AM notifies RM that job is done and releases all containers.

⭐ Key Points for Revision
  • YARN = Yet Another Resource Negotiator (Hadoop 2.x)
  • 4 main components: ResourceManager, NodeManager, ApplicationMaster, Container
  • ResourceManager = Master (1 per cluster)
  • NodeManager = Slave (1 per node, many per cluster)
  • ApplicationMaster = 1 per job/application
  • Container = resource bundle (CPU + RAM) for one task
  • YARN enables multi-tenancy: Spark and MapReduce can run on same cluster
Conclusion YARN revolutionized Hadoop by separating resource management from data processing. This enabled multiple processing frameworks to run on the same Hadoop cluster, maximizing resource utilization and making Hadoop a true general-purpose Big Data platform.

CAP Theorem: A distributed system can only guarantee 2 of 3: Consistency /\ / \ / \ / \ /_______ \ Availability Partition Tolerance CA - RDBMS (gives up partition tolerance) CP - HBase, MongoDB (gives up availability) AP - Cassandra, CouchDB (gives up consistency)

  • CAP Theorem: Consistency, Availability, Partition Tolerance (pick 2)

Conclusion NoSQL databases provide the flexibility, scalability, and performance needed for modern Big Data applications.
