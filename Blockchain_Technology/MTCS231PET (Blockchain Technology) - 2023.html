Q2 — Describe SHA-256 with a diagram6 marks
Answer

SHA-256 (Secure Hash Algorithm 256-bit) is a one-way cryptographic hash function that converts any input into a fixed 256-bit (32-byte) output called a digest or hash. It is used in Bitcoin mining, digital signatures, and data integrity checks.

Key properties:

  • Deterministic — same input always gives same output
  • Avalanche effect — changing 1 bit changes ~50% of output bits
  • Irreversible — cannot reverse-engineer the input from the hash
  • Collision resistant — virtually impossible to find two inputs with the same hash
Real example: SHA-256("hello") = 2cf24dba5fb0a30e26e83b2ac5b9e29e1b161e5c1fa7425e73043362938b9824 — always the same 64-character hex string regardless of where or when computed.
Q3 — What is a Fork? Discuss its types.6 marks
Answer

A Fork in blockchain occurs when the protocol rules change, causing the blockchain to split into two diverging paths. Forks happen due to software upgrades, disagreements in the community, or accidental divergence.

There is also a Accidental Fork (or temporary fork) — when two miners find a valid block simultaneously. The network resolves this by choosing the longest chain within a few blocks.

Q4 — Explain Proof-of-Stake (PoS) protocol with diagram6 marks
Answer

Proof-of-Stake (PoS) is a consensus mechanism where validators are chosen to create new blocks based on the amount of cryptocurrency they stake (lock as collateral), rather than computational power (as in PoW).

Key advantages over PoW: 99.95% less energy (Ethereum's merge reduced energy by this amount), faster finality, no need for expensive mining hardware. Ethereum switched to PoS in Sep 2022 (The Merge). Minimum stake = 32 ETH per validator.

Real example: If Alice stakes 64 ETH and Bob stakes 32 ETH, Alice has 2× the chance of being selected as the next block proposer.

Q7 — Explain Deterministic vs Non-Deterministic with examples6 marks
Answer

In the context of blockchain smart contracts, determinism is critical because all nodes must independently execute the same code and arrive at the identical result to reach consensus.

FeatureDeterministicNon-Deterministic
DefinitionSame input → same output always, on every nodeSame input → different outputs on different nodes/times
Used in Blockchain?Yes — required for consensusNo — breaks consensus
ExamplesHash functions (SHA-256), arithmetic (+, -, ×), string operationsRandom number generation (Math.random()), system time (Date.now()), external API calls
Smart contractx = a + b (always same)x = Math.random() (varies by node)
Fix for non-determinismUse oracles (Chainlink) for external data—
Example: A smart contract calculating "transfer 10% of balance" — this is deterministic. But a contract needing today's ETH price must use an Oracle, because each node getting the price directly would get slightly different values, breaking consensus.

In Hyperledger Fabric, chaincode is executed by endorsing peers first (determinism check), then the result is committed — this "execute-order-validate" model helps manage non-deterministic smart contract languages like Go or Java by comparing the RW sets.

Q9 — Ethereum Transaction Fee Calculation6 marks
Answer

Given: Bob sends Robin 20 ETH on Ethereum Network. Gas Limit = 21,000. Gas Price = 100 gwei. Priority Fee = 5 gwei. Gas used = 5 gas (assume all gas units used = 21,000 for standard ETH transfer).

Step-by-step calculation:

Gas Price = Base Fee + Priority Fee = 100 gwei (total) Gas Used = 21,000 units (standard ETH transfer) Transaction Fee = Gas Used × Gas Price = 21,000 × 100 gwei = 2,100,000 gwei = 2,100,000 / 1,000,000,000 ETH = 0.0021 ETH Priority Fee portion = 21,000 × 5 gwei = 105,000 gwei = 0.000105 ETH (goes to miner/validator)
Result: Bob pays 0.0021 ETH as transaction fees. Robin receives exactly 20 ETH. Bob's total deduction = 20 + 0.0021 = 20.0021 ETH from his wallet.

Key terms: 1 ETH = 10⁹ gwei = 10¹⁸ wei. Gas Limit is the maximum gas you allow. If the transaction uses less gas, the rest is refunded. Gas Price is how much you pay per unit of gas.

Q10 — ElGamal Encryption to encrypt a Plaintext block10 marks
Answer

Given: p = 11 (prime), e₁ = 2 (primitive root / generator), d = 3 (private key). Ciphertext: C₁ = 6, C₂ = 5.

Step 1 — Setup (Key Generation):

p = 11 (prime number) g = e₁ = 2 (primitive root / generator of Z*p) d = 3 (recipient's private key) Public key component: e₂ = g^d mod p = 2^3 mod 11 = 8 mod 11 = 8 Public Key = (p=11, g=2, e₂=8) Private Key = d = 3

Step 2 — Decryption (Finding the plaintext from C₁=6, C₂=5):

ElGamal Decryption formula: M = C₂ × (C₁^d)^(-1) mod p Step A: Compute C₁^d mod p = 6^3 mod 11 = 216 mod 11 = 216 - 19×11 = 216 - 209 = 7 C₁^d mod p = 7 Step B: Find modular inverse of 7 mod 11 Need x such that 7x ≡ 1 (mod 11) Try: 7 × 8 = 56 = 5×11 + 1 = 1 (mod 11) ✓ So (C₁^d)^(-1) mod 11 = 8 Step C: Compute M M = C₂ × 8 mod 11 = 5 × 8 mod 11 = 40 mod 11 = 40 - 3×11 = 40 - 33 = 7 Plaintext M = 7
Answer: The plaintext M = 7. So the ciphertext (C₁=6, C₂=5) corresponds to the original message M = 7.

Verification (Encryption side): If M=7, choose random k: C₁ = g^k mod p. C₂ = M × e₂^k mod p. This confirms that ElGamal is an asymmetric encryption based on the Discrete Logarithm Problem.

Q11 — Write short notes on: Double Spending, Uncle Block, Stale Block, Orphan Block10 marks
Answer

(a) Double Spending Money

Double spending is the attempt to spend the same digital currency twice. Because digital data can be copied, without a blockchain, nothing stops a malicious user from duplicating a transaction. Blockchain prevents this by requiring network consensus before a transaction is confirmed — once recorded in a block with sufficient confirmations, it cannot be reversed.

Example: Alice has 1 BTC. She broadcasts two transactions simultaneously — one sending 1 BTC to Bob, one sending 1 BTC to Charlie. The network will confirm only one; the other is rejected.

(b) Uncle Block (Ethereum specific)

When two miners find a valid block at the same time, the network chooses one as the main chain block. The other valid-but-excluded block is called an Uncle Block (or Ommer Block) in Ethereum. Unlike Bitcoin (where these are simply orphaned), Ethereum includes uncle blocks in the chain and rewards uncle miners partially — improving fairness and security.

(c) Stale Block

A stale block is a valid block that was mined correctly but not included in the main chain because another block at the same height was accepted first. The miner receives no reward. Stale blocks occur due to network propagation delays — by the time the block reaches the network, another block has already been accepted.

(d) Orphan Block

An orphan block is a block whose parent block (previous block) is unknown to the receiving node — it hasn't received the parent yet due to network delay. Orphan blocks are temporarily stored until their parent arrives. If no parent ever arrives, they are discarded. (Note: in casual usage, "orphan" is sometimes used interchangeably with "stale", but technically they are different.)

Q12 — Explain how PoW Consensus is used in Bitcoin Mining10 marks
Answer

Proof-of-Work (PoW) is the original consensus mechanism used by Bitcoin. Miners compete to solve a computationally expensive mathematical puzzle, and the winner gets to add the next block and earn the block reward.

The Puzzle: Find a value called the Nonce such that when combined with the block data, SHA-256(SHA-256(block_header)) produces a hash starting with a required number of leading zeros (the target difficulty).

SHA-256( SHA-256( Version | PrevHash | MerkleRoot | Timestamp | Difficulty | Nonce ) ) must be ≤ Target Value Example Target (simplified): 0000000000000000029... (many leading zeros = harder puzzle) Miner tries: Nonce = 0, 1, 2, 3, ... billions of times until hash matches target

Key properties of PoW:

  • Asymmetric: Hard to find nonce (billions of tries), but easy to verify (just hash once)
  • 51% attack: An attacker needs 51%+ of total hashrate to rewrite history — extremely costly
  • Energy intensive: Bitcoin consumes ~127 TWh/year (comparable to Norway)
  • Block reward halves every 210,000 blocks (~4 years) — current reward is 3.125 BTC

Q14 — Explain Hyperledger Transaction Flow with Diagram10 marks
Answer

Hyperledger Fabric uses an Execute → Order → Validate (EOV) model. This is different from Ethereum's Order → Execute model. The EOV model enables parallel chaincode execution and flexible trust models.

Detailed steps:

  1. Client creates transaction proposal — specifies which chaincode function to call, with args, and sends to endorsing peers.
  2. Endorsing peers execute chaincode — simulate transaction against current state, generate a Read-Write (RW) set, sign the result. No ledger update yet.
  3. Client collects endorsements — verifies all required peers have signed (per endorsement policy). If not enough signatures, abort.
  4. Client sends to Ordering Service — the transaction (with signatures and RW set) is sent to orderers (Raft/Kafka based).
  5. Orderer creates block — orders transactions, batches them into a new block, delivers to all committing peers.
  6. Peers validate and commit — each peer checks endorsement policy compliance, checks for RW set conflicts (MVCC), then appends valid transactions to the blockchain and updates the world state database.
  7. Event notification — peers emit events; the client app learns if its transaction was committed or invalidated.
Key advantage: Because endorsement (execution) happens before ordering, Hyperledger Fabric can handle much higher throughput than Ethereum — thousands of transactions per second vs. ~15 TPS for Ethereum mainnet.
